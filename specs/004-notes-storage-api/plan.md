# Implementation Plan: Notes & Storage API

**Branch**: `004-notes-storage-api` | **Date**: 2026-03-04 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/004-notes-storage-api/spec.md`

## Summary

Build backend CRUD API for `Storage` (hierarchical named folders) and `Note` (titled text content within a Storage) entities, backing the existing frontend notes page which currently uses in-memory mock data. Add two new NestJS modules (`StorageModule`, `NoteModule`) following the exact `BlockModule` pattern, extend the Prisma schema with two new models (self-referential `Storage` and `Note`), run a migration, and provide full test coverage (unit, integration, E2E) at all three layers already established in the project.

## Technical Context

**Language/Version**: TypeScript 5.9 (strict mode, ES2023 target, nodenext modules)
**Primary Dependencies**: NestJS 11, Prisma 7 + pg adapter, class-validator, class-transformer, @nestjs/swagger, Vitest 3, Playwright 1.50
**Storage**: PostgreSQL via Prisma 7 — two new tables: `storages` (self-referential), `notes`
**Testing**: Vitest (unit + integration), Playwright (E2E) — matching existing three-layer test setup
**Target Platform**: Linux server (Node >=18, deployed as Docker/Vercel Lambda)
**Project Type**: REST web-service (NestJS API within Turborepo monorepo)
**Performance Goals**: All CRUD responses < 500ms under normal load (SC-002)
**Constraints**: All queries scoped to `userId`; no pagination required for MVP; hard deletes (no soft-delete for Storage/Note, unlike Block)
**Scale/Scope**: MVP — single user data, no team features, no real-time
