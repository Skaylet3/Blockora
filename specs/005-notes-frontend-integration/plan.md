## Summary

## Implementation Strategy

### Phase 1: API Client Modules (Foundation)
Create `storages.api.ts` and `notes.api.ts` in `shared/api/`. These are pure TypeScript modules with no React ‚Äî independently testable. Write unit tests using `vi.mock('@/shared/api/http-client')`.

### Phase 2: NotesPage Integration
Replace `initialStorages` and `initialNotes` mock data with API calls in `notes-page.tsx`:
1. Add `loadingStorages` + `storagesError` state; load storages on mount via `useEffect`
2. Add `notesCache` state (`Map<string, NoteItem[]>`); load notes when a storage is opened
3. Convert all mutation handlers (`confirmAddStorage`, `handleDeleteStorage`, `handleCreateNote`/`handleSaveNote`, `handleDeleteNote`) to async API calls
4. Add loading indicator for initial load
5. Add inline error state for load failure
6. Disable Save/action buttons during in-flight requests

### Phase 3: Tests
Add component smoke tests for `NotesPage` verifying:
- Loading state renders without crashing
- Mock storages are NOT present (empty state) when API returns `[]`
- Error state renders when API rejects

## Key Design Decisions (from research.md)

1. **`useEffect` + `useState` pattern** ‚Äî matches `BlocksClient`; no new library needed
2. **Two separate API files** ‚Äî `storages.api.ts` + `notes.api.ts`; mirrors `blocks.api.ts`
3. **Per-storage note loading** ‚Äî avoids loading all notes on mount; cached in `Map`
4. **`toast.error()` for mutations, inline error for initial load** ‚Äî matches existing UX patterns
5. **Client-side cascade on storage delete** ‚Äî mirrors backend cascade without re-fetch
6. **`expanded` state stays UI-only** ‚Äî not persisted to backend; all loaded storages start collapsed
7. **No optimistic updates** ‚Äî UI only changes on successful API response
